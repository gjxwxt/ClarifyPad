# 03. 初版 PRD：Pre-submit AI Input Layer v0.1

## 1. 产品名称

暂定名：**思桥 ClarifyPad**

备选：
- 输入中转台
- AI 提交前助手
- PromptPad
- ClearEnter
- Thought2Text

## 2. 产品定位

**思桥是一款跨应用的 AI 输入中间层。用户在任意输入框中按快捷键唤起浮层，输入粗糙想法后，系统在提交前将其整理为清晰、可执行、语气合适的文本，并一键插入当前光标位置。**

## 3. 产品目标

### 3.1 v0.1 目标

验证用户是否愿意在正式发送前多经过一个 AI 中间层。

### 3.2 v0.1 不做什么

- 不做完整输入法；
- 不做语音输入；
- 不做长文编辑器；
- 不做团队知识库；
- 不做复杂 Agent 自动执行；
- 不做全量屏幕读取；
- 不在未确认时保存用户所有输入。

## 4. 目标平台

v0.1 推荐优先级：

1. **macOS 桌面端**
   - 知识工作者集中；
   - 全局快捷键、辅助功能、剪贴板插入相对成熟；
   - 适合快速验证。
2. Windows 作为 v0.2 或 v0.3。
3. 浏览器插件作为另一路径，可后续验证。

## 5. 核心用户故事

### 用户故事 1：发送前精简

作为一个工作 IM 高频用户，  
我希望在发消息前把啰嗦草稿变得简洁，  
以便对方能更快理解我的意思。

验收标准：
- 用户能用快捷键打开弹窗；
- 输入原始草稿后，选择“精简”；
- 系统在 3 秒内给出结果；
- 用户可一键插入当前输入框。

### 用户故事 2：想法变任务

作为产品/项目负责人，  
我希望把零散想法变成任务说明，  
以便团队知道背景、目标、交付物和验收标准。

验收标准：
- 输出包含标题、背景、任务、验收标准、待确认问题；
- 用户可选择插入完整版本或只插入精简版本。

### 用户故事 3：缺口提醒

作为表达不够确定的用户，  
我希望 AI 能告诉我这段话还缺什么，  
以便我在发出去前补充关键信息。

验收标准：
- 系统在输出旁显示 1-5 条缺口提醒；
- 缺口提醒必须短、小、可操作；
- 用户点击提醒后可快速补充。

### 用户故事 4：确认记录

作为长期用户，  
我希望保存每次确认插入的内容，  
以便之后复用、回看、总结自己的表达习惯。

验收标准：
- 每次点击“确认插入”后自动生成一条历史；
- 历史包含原始输入、最终文本、时间、模式、目标 App；
- 用户可以搜索历史；
- 用户可以删除历史；
- 默认不保存密码框/敏感应用内容。

## 6. 功能范围

### 6.1 v0.1 必做功能

#### F1. 全局快捷键唤起

- 默认快捷键：`Option + Space` 或 `Ctrl + Space`，需避免与系统冲突；
- 支持用户自定义；
- 当前有输入焦点时，在光标附近显示浮层；
- 无法定位光标时，居中显示。

#### F2. 输入浮层

浮层包含：
- 原始输入区；
- 模式选择；
- 生成按钮；
- 结果区；
- 缺口提醒区；
- 确认插入按钮；
- 取消按钮。

基础交互：
- `Enter`：生成/刷新归纳；
- `Cmd/Ctrl + Enter`：确认插入；
- `Shift + Enter`：换行；
- `Esc`：关闭；
- `Tab`：切换模式/结果。

#### F3. AI 处理模式

v0.1 内置 6 个模式：

1. **智能整理**
   - 自动判断场景，输出最适合的版本。
2. **精简**
   - 去掉废话，保留重点。
3. **深化**
   - 补足逻辑、背景、影响和建议。
4. **任务化**
   - 转成任务描述、验收标准、待确认问题。
5. **友好回复**
   - 适合 IM/客户回复，语气自然。
6. **Prompt 优化**
   - 转成适合给 AI 的结构化 prompt。

#### F4. 多版本输出

每次生成最多给 3 个版本：

- 直接可发版
- 更短版
- 更完整/更可执行版

每个版本提供：
- 复制
- 插入
- 继续修改
- 收藏为模板

#### F5. 缺口提醒

根据输入内容判断是否缺少：
- 对象：给谁看？
- 背景：为什么要说？
- 目标：希望对方做什么？
- 时间：什么时候？
- 标准：怎么算完成？
- 语气：要强硬、委婉、正式还是轻松？
- 材料：是否需要附链接/截图/数据？

显示方式：
- 结果区下方的小字；
- 最多 5 条；
- 每条不超过 20 个字；
- 点击后进入补充输入状态。

#### F6. 一键插入

插入逻辑优先级：
1. 使用系统辅助功能 API 直接插入；
2. 失败时复制到剪贴板并模拟粘贴；
3. 再失败时只复制，并提示用户手动粘贴。

插入后：
- 弹窗关闭；
- 保存确认记录；
- 可短时间内撤销。

#### F7. 确认记录

保存字段：
- id
- created_at
- source_app
- input_raw
- output_final
- mode
- gap_hints
- user_edited_after_insert（v0.2 可做）
- tags
- is_favorite

历史页功能：
- 列表；
- 搜索；
- 按模式筛选；
- 复制历史输出；
- 删除记录；
- 导出 Markdown / JSON。

#### F8. 隐私设置

基础设置：
- 是否保存原始输入；
- 是否保存最终输出；
- 黑名单 App；
- 敏感词提醒；
- 默认模型供应商；
- 是否使用云端模型；
- 删除全部历史。

默认策略：
- 不在密码输入框启用；
- 黑名单包含密码管理器、银行、加密钱包等；
- 用户未点击确认插入时，不保存内容；
- 明确展示“本次将发送到哪个模型服务”。

### 6.2 v0.1 可选功能

- 截图/读屏上下文；
- 团队模板；
- 语音输入；
- 移动端输入法；
- 浏览器插件；
- 自动识别聊天对象；
- 企业 SSO / 审计。

这些功能会增加权限、隐私和开发复杂度，不建议作为第一版核心。

## 8. AI 输出策略

### 8.1 输入分析维度

AI 先判断：
- 场景：回复 / 任务 / 需求 / prompt / 总结 / 反馈 / 其他；
- 受众：同事 / 上级 / 下级 / 客户 / AI / 未知；
- 情绪：中性 / 焦急 / 不满 / 委婉 / 强硬；
- 完整度：是否缺目标、背景、时间、标准；
- 推荐处理方式。

### 8.2 输出格式

推荐统一 JSON 返回，前端再渲染：

```json
{
  "detected_scene": "task",
  "recommended_mode": "任务化",
  "versions": [
    {
      "title": "直接可发版",
      "text": "请在本周五前完成积分页面的规则说明优化……",
      "reason": "保留目标、时间和交付物"
    }
  ],
  "gap_hints": [
    "截止时间还不明确",
    "需要补充验收标准"
  ],
  "memory_tags": ["产品需求", "任务分配", "积分系统"]
}
```

## 10. 权限要求

macOS v0.1 需要：
- Accessibility：全局快捷键、读取/插入选区、模拟粘贴；
- Input Monitoring：视实现方式而定，尽量避免；
- Screen Recording：v0.1 不建议默认申请，除非做屏幕上下文；
- Network：云端模型调用；
- Local storage：历史记录。

## 11. 非功能需求

### 性能

- 弹窗唤起：< 150ms；
- 首 token 返回：< 1.5s；
- 一次生成完成：目标 < 3s，复杂任务 < 6s；
- 插入成功率：首批目标 > 90% 常用 App。

### 安全

- 本地加密历史；
- 支持全部删除；
- 明确模型调用提示；
- 黑名单 App；
- 不读取密码框；
- 不在用户未确认时保存内容。

### 可靠性

- 插入失败时自动 fallback 到复制；
- 模型失败时保留用户输入；
- 网络失败时可用本地模板生成基础结构；
- 历史写入失败不得影响插入。

## 13. MVP 验证实验

### 实验 1：Figma 原型 + Wizard of Oz

不写完整客户端，先让用户在一个模拟浮层里输入，后台人工/LLM 生成结果，观察是否愿意用。

### 实验 2：Raycast / Alfred 插件原型

降低跨 App 开发成本，先验证：
- 选中文本或输入草稿；
- AI 改写；
- 粘贴回当前 App；
- 历史记录。

### 实验 3：macOS Electron/Tauri 原型

实现真实全局浮层和插入，验证技术体验。

## 15. 关键开放问题

1. 用户是否愿意为了“表达更清楚”多按一次快捷键？
2. 缺口提醒会不会被认为打扰？
3. 保存确认历史是否能带来长期价值？
4. 没有屏幕上下文时，输出是否足够好？
5. 跨 App 插入稳定性是否能达到日常可用？
6. 和系统级 Writing Tools 相比，用户为什么要用我们？
