# 06. Prompt 与交互样例

## 1. 系统 Prompt 草案

你是一个“提交前表达整理器”。用户会输入一段粗糙、零散、口语化的文字。你的任务不是炫技写作，而是在用户真正发送前，把内容变成清晰、可执行、语气合适的表达。

你需要：
1. 判断用户意图和场景；
2. 输出 1-3 个可直接使用的版本；
3. 指出还缺少哪些关键信息；
4. 保留用户真实意图，不编造事实；
5. 如果信息不足，使用“待确认”表达，而不是擅自补充；
6. 输出 JSON。

## 2. 输出 JSON Schema

```json
{
  "scene": "reply | task | requirement | prompt | feedback | summary | other",
  "mode_used": "smart | concise | deepen | task | friendly_reply | prompt",
  "versions": [
    {
      "name": "直接可发版",
      "text": "",
      "best_for": "",
      "risk": ""
    }
  ],
  "gap_hints": [
    {
      "hint": "",
      "why_it_matters": ""
    }
  ],
  "suggested_followups": [],
  "memory_tags": []
}
```

## 3. 样例 A：任务化

### 用户原始输入

> 那个积分的页面需要改一下，用户现在看不太明白规则，而且运营那边应该也想自己配一些文案，你帮我想一下怎么跟研发说。

### 输出

**直接可发版**

> 我们需要优化一下积分页面的规则说明，主要目标是让用户更清楚地理解积分获取、消耗和有效期规则。  
> 建议本次先做两件事：  
> 1. 前端页面增加更清晰的规则说明区域；  
> 2. 后台支持运营配置规则文案，避免后续每次改文案都需要研发发版。  
>   
> 可以先评估一下实现成本吗？如果工作量可控，我建议排进下一版需求。

**任务化版**

> 任务：优化积分页面规则说明  
> 背景：当前用户对积分规则理解不清晰，可能影响积分功能使用。  
> 目标：让用户清楚了解积分获取、消耗、有效期等规则。  
> 需求：  
> - 前端增加规则说明展示区域；  
> - 后台支持运营配置规则文案；  
> - 文案调整不依赖研发发版。  
> 待评估：实现成本、是否影响现有积分逻辑、是否需要灰度。  
> 验收标准：用户能在积分页面看到完整规则说明；运营可在后台修改展示文案。

**还可以补充**
- 期望什么时候上线？
- 规则文案由谁提供？
- 是否只改文案，还是也改积分逻辑？

## 4. 样例 B：把情绪表达变成建设性反馈

### 用户原始输入

> 这个页面真的太乱了，入口太多了，用户肯定找不到，之前说要改那个按钮也没改，感觉这样上线不太行。

### 输出

**克制反馈版**

> 我有点担心当前页面的信息层级。现在入口比较多，主操作不够突出，用户可能难以快速找到核心路径。  
> 另外，上次讨论过的按钮调整似乎还没有体现。如果这个问题不解决，直接上线可能会影响转化和用户理解。  
> 建议我们在上线前至少确认两点：  
> 1. 主按钮是否需要按之前方案调整；  
> 2. 页面入口是否可以做一次优先级收敛。

**还可以补充**
- 这是发给设计、产品还是研发？
- 是否有数据支持“找不到入口”？
- 上线时间是否已经确定？

## 5. 样例 C：Prompt 优化

### 用户原始输入

> 帮我分析这些用户反馈，看看产品哪里有问题，然后给点建议。

### 输出

**Prompt 优化版**

> 你是一名资深产品分析师。请基于我提供的用户反馈，分析产品当前存在的问题，并给出可执行的优化建议。  
>   
> 请按以下结构输出：  
> 1. 反馈主题分类：将用户反馈按问题类型归类；  
> 2. 高频问题：指出出现频率最高、影响最大的 3-5 个问题；  
> 3. 根因分析：分析这些问题可能对应的产品设计、流程或认知成本原因；  
> 4. 优先级建议：按用户影响和实现成本排序；  
> 5. 具体改进方案：每条建议说明目标、改动点、预期效果；  
> 6. 仍需补充的信息：列出为了判断更准确还需要哪些数据。  
>   
> 输出要求：  
> - 不要泛泛而谈；  
> - 每个结论都尽量引用具体用户反馈；  
> - 最后给出一个优先级表格。

**还可以补充**
- 用户反馈原文在哪里？
- 是否需要按用户类型分类？
- 是否有业务指标或转化数据？

## 6. 交互细节建议

### Enter 的设计

- Enter：执行“归纳/生成”
- Shift + Enter：换行
- Cmd/Ctrl + Enter：确认插入
- Esc：退出
- Cmd/Ctrl + 1/2/3：插入对应版本

这会让“Enter = 思考归纳”成为产品心智。

### 缺口提醒的文案风格

不要写：
- “你的描述不完整”
- “请补充更多信息”
- “你没有说明目标”

应该写：
- “希望对方做什么？”
- “有没有截止时间？”
- “怎么算完成？”
- “这是发给谁？”
- “需要更委婉吗？”

## 7. 模式说明

### 智能整理

适合默认场景。AI 自动判断用户是要回复、布置任务、写需求、写 prompt 还是反馈问题。

### 精简

适合 IM、客户回复、领导同步。

### 深化

适合需求说明、方案讨论、复盘。

### 任务化

适合发给研发、设计、运营、外包、客服团队。

### 友好回复

适合客户、同事、群聊，强调自然和低冲突。

### Prompt 优化

适合发给 AI 工具，强调角色、任务、输入、输出、约束。
