# 思桥 ClarifyPad：方案、需求分析与初版 PRD

# Pre-submit AI Input Layer：文字输入前的 AI 中间层

> 暂定中文名：思桥 / 输入中转台 / ClarifyPad  
> 版本：v0.1 方案与初版 PRD  
> 日期：2026-05-04

## 一句话概括

在任何工作软件的光标处，用快捷键唤起一个轻量输入弹窗。用户先用自然、零散、口语化的文字把想法写进去，AI 在真正提交前完成“归纳、精简、深化、追问、结构化、风格适配”，最后一键插入当前输入框，并把用户确认过的内容沉淀为个人表达与工作记忆。

## 这个文件夹包含什么

- `01_market_and_positioning.md`：竞品与机会判断
- `02_requirements_analysis.md`：用户、场景、痛点、需求拆解
- `03_prd_v0_1.md`：初版 PRD
- `04_mvp_roadmap.md`：MVP 路线、里程碑、验证指标
- `05_technical_feasibility.md`：技术实现建议与风险
- `06_prompt_and_interaction_samples.md`：提示词、交互样例、输出样例
- `07_source_links.md`：调研来源链接

## 关键判断

这个方向不是“没人做”，而是“通用 AI 改写已经有人做，文字输入前的思考中间层仍有差异化空间”。

真正值得做的不是又一个“把文字改得更正式”的工具，而是一个：
1. 跨 App 的输入前弹窗；
2. 面向工作执行的结构化器；
3. 会提醒信息缺口的澄清器；
4. 会记住每次确认内容的个人表达与任务记忆层；
5. 能把 Enter / 确认动作变成一次“思考提交”的产品。

## 推荐 MVP 定位

第一版不要做成完整输入法，也不要做成 ChatGPT 外壳。建议聚焦：

**“工作聊天、任务描述、AI Prompt、邮件/工单回复的提交前优化器”。**

核心路径：

快捷键 → 光标旁弹窗 → 输入草稿 → AI 输出 3 个结果：
- 可直接发送版
- 更简洁版
- 更可执行版

同时给出：
- 信息缺口提醒
- 可追问问题
- 一键插入
- 保存确认记录

# 01. 市场与定位分析

## 1. 背景

用户已经习惯在工作流中频繁输入文字：微信/飞书/Slack、邮件、Notion、Jira、Linear、飞书文档、ChatGPT、Claude、CRM、客服系统等。AI 让“写得更好”变得便宜，但当前主流工具大多集中在两类：

1. **语音输入 + 自动整理**：用户说话，系统转文字并清洗口语。
2. **选中文本后改写**：已有一段文字，选中后执行润色、翻译、总结、变短、变正式。

你的想法切入的是第三类：**打字时，在真正提交前，插入一个“思考-归纳-确认”的中间层**。

这类产品的核心价值不是提高打字速度，而是降低“想法到可执行表达”的损耗。

## 2. 竞品观察

### 2.1 语音输入类

**闪电说**  
特点：语音输入、语音修改、屏幕上下文理解、本地记忆、自定义技能、结果自动填入。它的关键优势是把“语音 + 技能执行”结合起来，不只是转写。

**Wispr Flow**  
特点：把自然语音转成格式化文本，去除 filler words / typos，支持个人词典和语音片段库。它证明了“粗糙输入 → 清晰输出”这个需求真实存在。

这类产品的启发：用户愿意把未经组织的表达交给 AI 整理。但它们的起点是语音，而不是工作环境中更常见、更隐蔽的文字输入。

### 2.2 系统级 / 输入级写作工具

**Apple Intelligence Writing Tools**  
Apple 已经把 rewrite、proofread、summarize 做成系统级能力，能在 Mail、Notes、Pages 和第三方应用中使用。系统级能力会压缩“通用润色工具”的空间。

**Microsoft Notepad AI**  
Windows Notepad 的 Rewrite / Summarize / Write 说明：基础编辑器也在加入 AI 写作能力。微软提供改写、调整语气、长度、生成多版本等功能。

这类产品的启发：大厂正在把 AI 写作下沉到操作系统和基础应用层。任何只做“润色/改写/总结”的产品都会面临平台级竞争。

### 2.3 AI 写作助手 / 跨 App 改写

**Grammarly**  
强调清晰、专业、语气、品牌风格、受众理解等能力，适合英文写作与企业场景。

**RewriteBar / Selected / Rewrait / WritingTools 等**  
这些工具大多围绕“选中已有文本 → 快捷键 → 改写/翻译/纠错/总结 → 替换文本”展开。开源项目 WritingTools 也已经覆盖了跨 App、快捷键、改写、摘要、本地/云端模型等能力。

这类产品的启发：跨 App 调用 AI 写作已经成立，但多数仍是“处理已有文本”，较少把“草稿输入、缺口提醒、思考归纳、确认沉淀”做成完整闭环。

## 3. 市场机会判断

### 3.1 有机会，但不适合做成泛化润色工具

如果定位为“按快捷键，把文字变正式”，竞争会非常激烈：
- 操作系统会自带；
- Grammarly / Superhuman 这类成熟玩家会覆盖；
- 开源工具和插件会快速复制；
- 大模型客户端本身也会做。

真正的机会在于：

**把它从“写作工具”升级为“工作表达的提交前决策层”。**

也就是：不是帮用户写得漂亮，而是帮用户在发出去之前想清楚。

### 3.2 推荐定位

> 一个面向工作流的 AI 输入中间层：把用户的粗糙输入变成清晰、可执行、上下文合适、可沉淀的表达。

这个定位下，产品不只做改写，而是做五件事：

1. **整理**：把零散想法归纳成清楚表达。
2. **压缩**：把长话变短，适合 IM / 工单 / 邮件。
3. **深化**：补足逻辑、目标、背景、验收标准。
4. **追问**：指出信息缺口，提示用户继续补充。
5. **沉淀**：保存用户确认过的表达，形成个人工作语料和偏好。

## 4. 可差异化切口

### 切口 A：工作执行语义，而不是文学润色

用户输入：

> 这个功能感觉要做一下，就是客户能看到自己的积分，但是规则可能要清楚一点，最好运营那边也能配。

普通改写工具会输出更通顺的句子。  
本产品应输出：

- 需求描述
- 背景
- 用户故事
- 验收标准
- 待确认问题
- 建议发给研发/运营的版本

这才是工作价值。

### 切口 B：Enter = 归纳总结 / 确认沉淀

你提到“点击 Enter 键就是对自己输入的一种归纳总结”，这是一个有产品辨识度的交互隐喻。

可以设计为：
- `Enter`：生成/刷新归纳
- `Cmd/Ctrl + Enter`：确认并插入
- `Shift + Enter`：换行
- 每次确认都进入“确认记录”
- 系统从确认记录中学习用户风格、常用任务、表达结构

### 切口 C：缺口提醒

不是只给答案，而是在答案旁边用小字提示：
- 还缺少对象：发给谁？
- 还缺少时间：什么时候完成？
- 还缺少标准：怎么算完成？
- 还缺少边界：不做什么？
- 还缺少语气：正式/强硬/委婉/轻松？

这个功能会把它从“AI 帮我写”变成“AI 帮我想清楚”。

### 切口 D：跨 App 轻量浮层

不要让用户复制到 ChatGPT。应该在当前光标位置附近出现浮层，处理完直接插入。关键是“不打断工作流”。

## 5. 风险

### 5.1 平台风险

macOS / Windows 对全局快捷键、光标定位、选区读取、文本插入、屏幕读取都有权限要求。不同 App 的输入框实现不同，兼容性会很复杂。

### 5.2 大厂挤压

Apple、Microsoft、Grammarly 等会覆盖通用写作能力。必须避免做成“弱版系统 Writing Tools”。

### 5.3 隐私与信任

这是一个全局输入工具。用户会天然担心：
- 是否记录所有输入？
- 是否上传敏感内容？
- 是否读取屏幕？
- 公司是否允许使用？

隐私设计必须成为产品卖点，而不是后补说明。

### 5.4 成本与延迟

如果每次输入都调用大模型，延迟和成本都会影响体验。需要：
- 本地小模型处理轻量任务；
- 云端大模型处理复杂任务；
- 模板化任务减少 token；
- 缓存用户风格和常用结构。

## 6. 结论

这个方向值得做，但 MVP 的生死点不是模型能力，而是：

1. 能否在任何输入框里足够稳定地唤起、插入；
2. 能否比用户“复制到 ChatGPT 再复制回来”快 5 倍；
3. 能否让输出更可执行，而不只是更漂亮；
4. 能否通过确认记录形成越用越顺手的个人化体验。

推荐把第一版做成“面向工作消息和任务描述的提交前 AI 中间层”，而不是泛 AI 写作助手。

# 02. 深入需求分析

## 1. 目标用户

### 1.1 首批用户

推荐优先服务以下人群：

1. **产品经理 / 项目经理**
   - 经常把模糊想法变成需求、任务、会议结论。
   - 痛点是表达不够清楚，容易被研发/设计/运营误解。

2. **运营 / 市场 / 销售**
   - 经常写私域消息、客户回复、活动说明、跟进记录。
   - 痛点是信息多、场景碎、需要语气合适。

3. **管理者 / 创业者**
   - 经常布置任务、反馈问题、做决策同步。
   - 痛点是临时想法多，但发出去前需要清晰、克制、可执行。

4. **客服 / 客成 / 售后**
   - 高频回复、需要统一口径。
   - 痛点是重复性强，但又不能机械。

5. **AI 高频用户**
   - 经常给 ChatGPT / Claude / Cursor / Manus 等工具写 prompt。
   - 痛点是 prompt 写得不够清楚导致输出不稳定。

### 1.2 不建议作为首批用户

1. 纯长文作者：他们更需要编辑器、知识管理和长文结构。
2. 只想纠错的人：Grammarly / 系统工具已足够。
3. 打字极快且表达能力强的人：除非有强记忆和自动化价值。
4. 对隐私极敏感且无法使用任何云端 AI 的企业用户：可以放到后续企业版。

## 2. 核心场景

### 场景 A：工作 IM 发送前优化

用户在飞书/微信/Slack 中想发一段话，但担心太啰嗦、太冲、太散。

流程：
1. 快捷键唤起输入中间层；
2. 输入粗糙想法；
3. AI 输出“直接发送版 / 委婉版 / 更短版”；
4. 用户点击确认；
5. 文本插入当前聊天框。

### 场景 B：把想法变成任务描述

用户想给团队布置任务，但只想到一些散点。

AI 输出：
- 任务标题
- 背景
- 要做什么
- 不做什么
- 截止时间
- 验收标准
- 待确认问题

### 场景 C：把吐槽变成建设性反馈

用户原始输入可能带情绪：

> 这个页面真的太乱了，用户肯定找不到入口，而且昨天说的那个按钮又没改。

AI 输出：
- 客观问题描述
- 影响
- 建议
- 需要对方确认的点
- 降低冲突的表达版本

### 场景 D：把粗糙 prompt 变成高质量 prompt

用户输入：

> 帮我分析这个用户反馈，看看产品哪里有问题。

AI 输出：
- 角色
- 任务
- 输入材料要求
- 输出格式
- 分析维度
- 约束条件
- 示例

### 场景 E：确认内容沉淀

每次用户点击“确认插入”后，系统记录：
- 原始输入
- 最终插入文本
- 场景
- 使用的处理方式
- 用户是否二次编辑
- 时间、目标 App、可选标签

长期形成：
- 常用表达模板
- 用户语气偏好
- 常见任务类型
- 个人 prompt 库
- 工作复盘素材

## 3. 用户痛点

### 3.1 表达前的痛点

- 想法是碎的，不知道怎么组织。
- 知道自己想表达什么，但写出来不够清楚。
- 害怕表达太冲、太弱、太啰嗦。
- 临时打字容易遗漏背景、时间、验收标准。
- 不想切到 ChatGPT，因为打断当前工作流。

### 3.2 表达后的痛点

- 发出去才发现别人理解错了。
- 任务没有验收标准，后续返工。
- 同样的问题反复解释，不能沉淀。
- 自己说过什么、确认过什么，散落在各个软件里。
- AI 工具帮忙写了，但没有持续学习自己的风格。

## 4. 需求分层

### 4.1 基础需求：可用

- 快捷键唤起
- 光标附近弹窗
- 支持输入文字
- 支持调用 AI 生成优化结果
- 一键插入当前输入框
- 记录确认历史

### 4.2 进阶需求：好用

- 多种输出模式：精简、正式、友好、任务化、Prompt 化、邮件化
- 输出前展示信息缺口
- 允许用户继续补充
- 支持多版本对比
- 支持撤销插入
- 支持常用模板
- 支持个人词典/术语

### 4.3 高阶需求：离不开

- 根据目标 App 自动推断场景
- 根据聊天对象/项目自动调整语气
- 从历史确认内容中学习表达偏好
- 自动生成个人工作复盘
- 支持团队模板和组织口径
- 支持本地模型和企业私有化

## 5. 用户旅程

### 第一次使用

1. 安装应用；
2. 授权辅助功能/输入权限；
3. 设置快捷键；
4. 打开任意输入框；
5. 按快捷键；
6. 输入一句粗糙想法；
7. 看到 AI 输出；
8. 一键插入；
9. 获得“确实省事”的首次价值。

### 高频使用

1. 用户不再思考“怎么组织语言”，先把想法丢进弹窗；
2. AI 自动识别是回复、任务、需求、prompt、总结；
3. 用户只做确认和小修；
4. 系统记录确认内容；
5. 下次输出更接近用户风格。

## 6. 核心用户价值

### 6.1 对个人

- 更快组织表达；
- 更少沟通误解；
- 更稳的语气和专业度；
- 形成个人表达资产；
- 把每天的零散想法沉淀下来。

### 6.2 对团队

- 任务描述更清晰；
- 会议和 IM 信息更可追踪；
- 口径更统一；
- 新人更容易写出合格表达；
- 减少反复追问和返工。

## 7. 核心假设

### 假设 1：文字输入场景下也存在“粗糙输入 → 清晰输出”的强需求

验证方式：
- 找 20 个知识工作者访谈；
- 观察他们在 IM / 邮件 / 任务系统里是否经常反复修改；
- 让他们试用原型，记录是否愿意替代 ChatGPT 复制粘贴。

### 假设 2：信息缺口提醒比单纯润色更有价值

验证方式：
- A/B 测试：只给改写 vs 改写 + 缺口提醒；
- 统计用户补充信息次数、最终插入率、满意度。

### 假设 3：确认记录能形成长期粘性

验证方式：
- 看用户是否回看历史；
- 是否从历史中复用表达；
- 是否愿意让系统基于历史学习偏好；
- 7 日/30 日留存是否因记忆功能提升。

## 8. MVP 成功标准

建议 v0.1 的成功标准：

- 日均唤起次数 ≥ 8 次/活跃用户；
- AI 生成后插入率 ≥ 50%；
- 生成后用户二次编辑比例 ≤ 40%；
- 7 日留存 ≥ 25%；
- 用户主观评分 ≥ 4/5；
- 用户至少能说出一个“以前需要复制到 ChatGPT，现在不用了”的场景。

# 03. 初版 PRD：Pre-submit AI Input Layer v0.1

## 1. 产品名称

暂定名：**思桥 ClarifyPad**

备选：
- 输入中转台
- AI 提交前助手
- PromptPad
- ClearEnter
- Thought2Text

## 2. 产品定位

**思桥是一款跨应用的 AI 输入中间层。用户在任意输入框中按快捷键唤起浮层，输入粗糙想法后，系统在提交前将其整理为清晰、可执行、语气合适的文本，并一键插入当前光标位置。**

## 3. 产品目标

### 3.1 v0.1 目标

验证用户是否愿意在正式发送前多经过一个 AI 中间层。

### 3.2 v0.1 不做什么

- 不做完整输入法；
- 不做语音输入；
- 不做长文编辑器；
- 不做团队知识库；
- 不做复杂 Agent 自动执行；
- 不做全量屏幕读取；
- 不在未确认时保存用户所有输入。

## 4. 目标平台

v0.1 推荐优先级：

1. **macOS 桌面端**
   - 知识工作者集中；
   - 全局快捷键、辅助功能、剪贴板插入相对成熟；
   - 适合快速验证。
2. Windows 作为 v0.2 或 v0.3。
3. 浏览器插件作为另一路径，可后续验证。

## 5. 核心用户故事

### 用户故事 1：发送前精简

作为一个工作 IM 高频用户，  
我希望在发消息前把啰嗦草稿变得简洁，  
以便对方能更快理解我的意思。

验收标准：
- 用户能用快捷键打开弹窗；
- 输入原始草稿后，选择“精简”；
- 系统在 3 秒内给出结果；
- 用户可一键插入当前输入框。

### 用户故事 2：想法变任务

作为产品/项目负责人，  
我希望把零散想法变成任务说明，  
以便团队知道背景、目标、交付物和验收标准。

验收标准：
- 输出包含标题、背景、任务、验收标准、待确认问题；
- 用户可选择插入完整版本或只插入精简版本。

### 用户故事 3：缺口提醒

作为表达不够确定的用户，  
我希望 AI 能告诉我这段话还缺什么，  
以便我在发出去前补充关键信息。

验收标准：
- 系统在输出旁显示 1-5 条缺口提醒；
- 缺口提醒必须短、小、可操作；
- 用户点击提醒后可快速补充。

### 用户故事 4：确认记录

作为长期用户，  
我希望保存每次确认插入的内容，  
以便之后复用、回看、总结自己的表达习惯。

验收标准：
- 每次点击“确认插入”后自动生成一条历史；
- 历史包含原始输入、最终文本、时间、模式、目标 App；
- 用户可以搜索历史；
- 用户可以删除历史；
- 默认不保存密码框/敏感应用内容。

## 6. 功能范围

### 6.1 v0.1 必做功能

#### F1. 全局快捷键唤起

- 默认快捷键：`Option + Space` 或 `Ctrl + Space`，需避免与系统冲突；
- 支持用户自定义；
- 当前有输入焦点时，在光标附近显示浮层；
- 无法定位光标时，居中显示。

#### F2. 输入浮层

浮层包含：
- 原始输入区；
- 模式选择；
- 生成按钮；
- 结果区；
- 缺口提醒区；
- 确认插入按钮；
- 取消按钮。

基础交互：
- `Enter`：生成/刷新归纳；
- `Cmd/Ctrl + Enter`：确认插入；
- `Shift + Enter`：换行；
- `Esc`：关闭；
- `Tab`：切换模式/结果。

#### F3. AI 处理模式

v0.1 内置 6 个模式：

1. **智能整理**
   - 自动判断场景，输出最适合的版本。
2. **精简**
   - 去掉废话，保留重点。
3. **深化**
   - 补足逻辑、背景、影响和建议。
4. **任务化**
   - 转成任务描述、验收标准、待确认问题。
5. **友好回复**
   - 适合 IM/客户回复，语气自然。
6. **Prompt 优化**
   - 转成适合给 AI 的结构化 prompt。

#### F4. 多版本输出

每次生成最多给 3 个版本：

- 直接可发版
- 更短版
- 更完整/更可执行版

每个版本提供：
- 复制
- 插入
- 继续修改
- 收藏为模板

#### F5. 缺口提醒

根据输入内容判断是否缺少：
- 对象：给谁看？
- 背景：为什么要说？
- 目标：希望对方做什么？
- 时间：什么时候？
- 标准：怎么算完成？
- 语气：要强硬、委婉、正式还是轻松？
- 材料：是否需要附链接/截图/数据？

显示方式：
- 结果区下方的小字；
- 最多 5 条；
- 每条不超过 20 个字；
- 点击后进入补充输入状态。

#### F6. 一键插入

插入逻辑优先级：
1. 使用系统辅助功能 API 直接插入；
2. 失败时复制到剪贴板并模拟粘贴；
3. 再失败时只复制，并提示用户手动粘贴。

插入后：
- 弹窗关闭；
- 保存确认记录；
- 可短时间内撤销。

#### F7. 确认记录

保存字段：
- id
- created_at
- source_app
- input_raw
- output_final
- mode
- gap_hints
- user_edited_after_insert（v0.2 可做）
- tags
- is_favorite

历史页功能：
- 列表；
- 搜索；
- 按模式筛选；
- 复制历史输出；
- 删除记录；
- 导出 Markdown / JSON。

#### F8. 隐私设置

基础设置：
- 是否保存原始输入；
- 是否保存最终输出；
- 黑名单 App；
- 敏感词提醒；
- 默认模型供应商；
- 是否使用云端模型；
- 删除全部历史。

默认策略：
- 不在密码输入框启用；
- 黑名单包含密码管理器、银行、加密钱包等；
- 用户未点击确认插入时，不保存内容；
- 明确展示“本次将发送到哪个模型服务”。

### 6.2 v0.1 可选功能

- 截图/读屏上下文；
- 团队模板；
- 语音输入；
- 移动端输入法；
- 浏览器插件；
- 自动识别聊天对象；
- 企业 SSO / 审计。

这些功能会增加权限、隐私和开发复杂度，不建议作为第一版核心。

# 04. MVP 路线图与验证计划

## 1. 总体策略

先验证“使用意愿”和“场景刚性”，再投入大量精力做系统级兼容。

不要一开始就做完整输入法。完整输入法涉及系统输入法框架、兼容性、候选栏、权限、隐私、跨平台等复杂问题，MVP 成本过高。

推荐路线：

1. 低保真原型验证需求；
2. Raycast / Alfred / 浏览器插件验证任务模式；
3. macOS 浮层客户端验证真实工作流；
4. Windows 和企业场景后置。

# 05. 技术可行性与架构建议

## 1. 技术目标

v0.1 要实现的不是完整输入法，而是：

> 全局快捷键唤起一个桌面浮层，用户输入后调用 AI，结果插入当前输入框，并保存确认历史。

这条链路技术可行，但跨 App 稳定性是主要难点。

## 2. 推荐架构

```
Desktop Client
  ├─ Global Hotkey Manager
  ├─ Floating Window Manager
  ├─ Text Input / Result UI
  ├─ Insert Engine
  ├─ Local History Store
  ├─ Privacy Guard
  └─ AI Orchestrator
        ├─ Prompt Templates
        ├─ Cloud LLM Provider
        ├─ Local LLM Provider
        └─ Output Parser
```

## 3. 客户端选型

### 方案 A：Swift + AppKit（推荐首版）

优点：
- macOS 权限、窗口、辅助功能集成最好；
- 浮层体验更自然；
- 插入、剪贴板、快捷键更稳定。

缺点：
- 跨平台复用弱；
- 团队需要 macOS 原生开发能力。

适合：认真做 macOS MVP。

### 方案 B：Tauri

优点：
- 体积小；
- 前端开发体验好；
- 后续跨平台较容易。

缺点：
- 系统级能力需要写 Rust/Swift/Objective-C 插件；
- 早期调试复杂度高。

适合：想兼顾跨平台但仍重视体积。

### 方案 C：Electron

优点：
- 开发快；
- 前端生态成熟；
- 跨平台方便。

缺点：
- 体积大；
- 系统级权限体验一般；
- 对一个输入增强工具来说可能显得重。

适合：快速 Demo，不建议长期首选。

## 4. 插入文本实现

### 优先级 1：Accessibility API

通过辅助功能获取当前焦点控件并插入文本。体验最好，但不同 App 兼容性不一。

### 优先级 2：剪贴板 + 模拟粘贴

流程：
1. 保存当前剪贴板；
2. 将结果写入剪贴板；
3. 模拟 `Cmd + V`；
4. 延迟恢复原剪贴板。

优点：兼容性强。  
缺点：可能影响用户剪贴板；需要处理失败和隐私提示。

### 优先级 3：只复制

当插入失败时，复制结果并提示用户手动粘贴。

## 5. 浮层定位

理想：定位在当前光标附近。  
实际：很多 App 不暴露 caret 坐标。

策略：
1. 能获取 caret bounds 时，显示在光标旁；
2. 获取不到时，显示在当前窗口中心；
3. 再失败时，显示在屏幕中心；
4. 允许用户固定浮层位置。

## 6. 隐私与安全

### 6.1 默认原则

- 不做全局键盘记录；
- 只有用户主动唤起后才读取输入；
- 未确认插入不保存；
- 默认不读取屏幕；
- 明示模型调用；
- 历史本地加密；
- 支持一键删除。

### 6.2 黑名单 App

内置黑名单：
- 1Password / Bitwarden / iCloud Keychain；
- 银行、证券、钱包类 App；
- 浏览器隐私模式可提示关闭保存；
- 企业用户可配置更多黑名单。

### 6.3 敏感内容检测

本地正则检测：
- 身份证、银行卡、手机号；
- API Key、Token；
- 密码样式；
- 私钥片段。

检测到后：
- 提示是否继续；
- 可默认切换到本地模型；
- 不保存历史。

## 7. AI 编排

### 7.1 模型选择

v0.1 建议支持：
- OpenAI-compatible API；
- Anthropic；
- Gemini；
- 本地 Ollama/OpenAI-compatible endpoint。

产品层面不绑定单一模型，允许用户自带 Key 或使用官方额度。

### 7.2 Prompt 策略

不要让用户每次都写复杂 prompt。系统根据模式组织 prompt：

输入：
- 原始内容；
- 当前模式；
- 目标 App；
- 用户偏好；
- 历史风格摘要（v0.2）；
- 输出 JSON schema。

输出：
- 场景判断；
- 多版本文本；
- 缺口提醒；
- 标签。

### 7.3 延迟优化

- 请求开始时立即展示 skeleton；
- 流式输出；
- 短文本用小模型；
- 常见模式 prompt 缓存；
- 历史风格摘要定期离线更新；
- 失败时降级到规则模板。

## 8. 本地数据

推荐 SQLite + SQLCipher 或系统 Keychain 加密 key。

表：
- history_items
- user_preferences
- prompt_templates
- app_rules
- model_configs

## 9. 兼容性测试清单

v0.1 优先测试：
- 飞书
- 微信
- Slack
- Chrome / Safari
- Gmail / Outlook Web
- Notion
- Linear / Jira
- VS Code / Cursor
- Apple Mail
- Notes

每个 App 测：
- 快捷键是否冲突；
- 浮层是否正常；
- 插入是否成功；
- 换行/提交行为是否受影响；
- 中文输入法状态是否正常。

## 10. 技术风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| 光标定位失败 | 浮层体验下降 | 提供窗口中心/固定位置 fallback |
| 插入失败 | 核心链路断裂 | Accessibility + 剪贴板 + 复制三层 fallback |
| 权限吓退用户 | 激活下降 | 首次使用时解释用途，分阶段申请权限 |
| 模型延迟高 | 用户放弃 | 流式输出、本地模板、缓存 |
| 隐私担忧 | 无法进入企业 | 本地优先、黑名单、透明模型调用 |
| 大厂功能覆盖 | 产品被替代 | 聚焦任务化、缺口提醒、历史记忆 |

# 06. Prompt 与交互样例

## 1. 系统 Prompt 草案

你是一个“提交前表达整理器”。用户会输入一段粗糙、零散、口语化的文字。你的任务不是炫技写作，而是在用户真正发送前，把内容变成清晰、可执行、语气合适的表达。

你需要：
1. 判断用户意图和场景；
2. 输出 1-3 个可直接使用的版本；
3. 指出还缺少哪些关键信息；
4. 保留用户真实意图，不编造事实；
5. 如果信息不足，使用“待确认”表达，而不是擅自补充；
6. 输出 JSON。

## 2. 输出 JSON Schema

```json
{
  "scene": "reply | task | requirement | prompt | feedback | summary | other",
  "mode_used": "smart | concise | deepen | task | friendly_reply | prompt",
  "versions": [
    {
      "name": "直接可发版",
      "text": "",
      "best_for": "",
      "risk": ""
    }
  ],
  "gap_hints": [
    {
      "hint": "",
      "why_it_matters": ""
    }
  ],
  "suggested_followups": [],
  "memory_tags": []
}
```

## 3. 样例 A：任务化

### 用户原始输入

> 那个积分的页面需要改一下，用户现在看不太明白规则，而且运营那边应该也想自己配一些文案，你帮我想一下怎么跟研发说。

### 输出

**直接可发版**

> 我们需要优化一下积分页面的规则说明，主要目标是让用户更清楚地理解积分获取、消耗和有效期规则。  
> 建议本次先做两件事：  
> 1. 前端页面增加更清晰的规则说明区域；  
> 2. 后台支持运营配置规则文案，避免后续每次改文案都需要研发发版。  
>   
> 可以先评估一下实现成本吗？如果工作量可控，我建议排进下一版需求。

**任务化版**

> 任务：优化积分页面规则说明  
> 背景：当前用户对积分规则理解不清晰，可能影响积分功能使用。  
> 目标：让用户清楚了解积分获取、消耗、有效期等规则。  
> 需求：  
> - 前端增加规则说明展示区域；  
> - 后台支持运营配置规则文案；  
> - 文案调整不依赖研发发版。  
> 待评估：实现成本、是否影响现有积分逻辑、是否需要灰度。  
> 验收标准：用户能在积分页面看到完整规则说明；运营可在后台修改展示文案。

**还可以补充**
- 期望什么时候上线？
- 规则文案由谁提供？
- 是否只改文案，还是也改积分逻辑？

## 4. 样例 B：把情绪表达变成建设性反馈

### 用户原始输入

> 这个页面真的太乱了，入口太多了，用户肯定找不到，之前说要改那个按钮也没改，感觉这样上线不太行。

### 输出

**克制反馈版**

> 我有点担心当前页面的信息层级。现在入口比较多，主操作不够突出，用户可能难以快速找到核心路径。  
> 另外，上次讨论过的按钮调整似乎还没有体现。如果这个问题不解决，直接上线可能会影响转化和用户理解。  
> 建议我们在上线前至少确认两点：  
> 1. 主按钮是否需要按之前方案调整；  
> 2. 页面入口是否可以做一次优先级收敛。

**还可以补充**
- 这是发给设计、产品还是研发？
- 是否有数据支持“找不到入口”？
- 上线时间是否已经确定？

## 5. 样例 C：Prompt 优化

### 用户原始输入

> 帮我分析这些用户反馈，看看产品哪里有问题，然后给点建议。

### 输出

**Prompt 优化版**

> 你是一名资深产品分析师。请基于我提供的用户反馈，分析产品当前存在的问题，并给出可执行的优化建议。  
>   
> 请按以下结构输出：  
> 1. 反馈主题分类：将用户反馈按问题类型归类；  
> 2. 高频问题：指出出现频率最高、影响最大的 3-5 个问题；  
> 3. 根因分析：分析这些问题可能对应的产品设计、流程或认知成本原因；  
> 4. 优先级建议：按用户影响和实现成本排序；  
> 5. 具体改进方案：每条建议说明目标、改动点、预期效果；  
> 6. 仍需补充的信息：列出为了判断更准确还需要哪些数据。  
>   
> 输出要求：  
> - 不要泛泛而谈；  
> - 每个结论都尽量引用具体用户反馈；  
> - 最后给出一个优先级表格。

**还可以补充**
- 用户反馈原文在哪里？
- 是否需要按用户类型分类？
- 是否有业务指标或转化数据？

## 6. 交互细节建议

### Enter 的设计

- Enter：执行“归纳/生成”
- Shift + Enter：换行
- Cmd/Ctrl + Enter：确认插入
- Esc：退出
- Cmd/Ctrl + 1/2/3：插入对应版本

这会让“Enter = 思考归纳”成为产品心智。

### 缺口提醒的文案风格

不要写：
- “你的描述不完整”
- “请补充更多信息”
- “你没有说明目标”

应该写：
- “希望对方做什么？”
- “有没有截止时间？”
- “怎么算完成？”
- “这是发给谁？”
- “需要更委婉吗？”

## 7. 模式说明

### 智能整理

适合默认场景。AI 自动判断用户是要回复、布置任务、写需求、写 prompt 还是反馈问题。

### 精简

适合 IM、客户回复、领导同步。

### 深化

适合需求说明、方案讨论、复盘。

### 任务化

适合发给研发、设计、运营、外包、客服团队。

### 友好回复

适合客户、同事、群聊，强调自然和低冲突。

### Prompt 优化

适合发给 AI 工具，强调角色、任务、输入、输出、约束。
